# 内容研究写作助手

本技能是你的写作伙伴，帮助你调研、列大纲、起草和润色内容，同时保持你的独特声音和风格。

## 适用场景

- 撰写博客、文章或通讯
- 创作教育内容或教程
- 起草思想领导力作品
- 调研并撰写案例分析
- 生成带有来源的技术文档
- 规范引用与参考文献写作
- 优化开头和引言
- 写作过程中分段获得反馈

## 功能说明

1. **协作大纲**：帮助你将想法结构化为连贯大纲
2. **调研助手**：查找相关信息并添加引用
3. **开头优化**：增强开头吸引力
4. **分段反馈**：你写每一段时都能获得点评
5. **风格保持**：维持你的写作风格和语气
6. **引用管理**：规范添加和格式化参考文献
7. **迭代完善**：多轮草稿协助你持续提升

## 使用方法

### 设置写作环境

为你的文章创建专属文件夹：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

创建草稿文件：
```
touch article-draft.md
```

在该目录下用 Claude Code 开始写作。

### 基本流程

1. **从大纲开始**：
```
帮我为[主题]创建一份文章大纲
```

2. **调研并添加引用**：
```
调研[具体主题]并为我的大纲添加引用
```

3. **优化开头**：
```
这是我的引言。帮我让开头更有吸引力。
```

4. **分段反馈**：
```
我刚写完“为什么重要”部分。请点评并给建议。
```

5. **润色完善**：
```
请检查全文的逻辑、清晰度和一致性。
```

## 操作指引

当用户请求写作协助时：

1. **理解写作项目**
   - 询问澄清问题：
     - 主题和主旨是什么？
     - 目标读者是谁？
     - 期望长度/格式？
     - 目标是什么？（教育、说服、娱乐、解释）
     - 有无现有调研或资料？
     - 偏好哪种写作风格？（正式、对话、技术）

2. **协作大纲**
   - 帮助结构化内容：
   ```markdown
   # 文章大纲：[标题]

   ## 开头
   - [开场白/故事/数据]
   - [为什么读者要关心]

   ## 引言
   - 背景与上下文
   - 问题陈述
   - 文章涵盖内容

   ## 主要部分

   ### 第一节：[标题]
   - 关键点A
   - 关键点B
   - 示例/证据
   - [需调研：具体主题]

   ### 第二节：[标题]
   - 关键点C
   - 关键点D
   - 需数据/引用

   ### 第三节：[标题]
   - 关键点E
   - 反方观点
   - 解决方案

   ## 结论
   - 总结要点
   - 行动号召
   - 结语

   ## 调研待办
   - [ ] 查找[主题]数据
   - [ ] 获取[概念]示例
   - [ ] [论点]的引用来源
   ```
   - **迭代大纲**：根据反馈调整，确保逻辑流畅，标记需深入调研部分

3. **调研**
   - 用户请求调研时：
     - 查找相关信息
     - 寻找权威来源
     - 摘录关键事实、引语和数据
     - 按需添加引用

   示例输出：
   ```markdown
   ## 调研：AI对生产力的影响

   主要发现：

   1. **生产力提升**：研究显示内容创作任务节省40%时间[1]
   2. **采用率**：67%的知识工作者每周使用AI工具[2]
   3. **专家引语**：“AI是增强而非取代人类创造力”——MIT的Jane Smith博士[3]

   引用：
   [1] 麦肯锡全球研究院（2024）《生成式AI的经济潜力》
   [2] Stack Overflow开发者调查（2024）
   [3] Smith, J. (2024). MIT科技评论访谈

   已添加到第二节大纲。
   ```

4. **优化开头**
   - 用户提供引言时，分析并增强：
   **当前开头分析**：
   - 优点：[亮点]
   - 可提升处：[改进点]
   - 情感影响：[现状与潜力]

   **建议替换**：
   选项1：大胆陈述
   > [示例]
   *理由：[解释]*

   选项2：个人故事
   > [示例]
   *理由：[解释]*

   选项3：惊人数据
   > [示例]
   *理由：[解释]*

   **钩子问题**：
   - 是否激发好奇？
   - 是否承诺价值？
   - 是否足够具体？
   - 是否契合受众？

5. **分段反馈**
   - 用户写每一节时，逐段点评：
   ```markdown
   # 反馈：[部分名称]

   ## 优点 ✓
   - [优点1]
   - [优点2]
   - [优点3]

   ## 改进建议

   ### 清晰度
   - [具体问题] → [建议修正]
   - [复杂句] → [更简明表达]

   ### 流畅性
   - [衔接问题] → [更好连接]
   - [段落顺序] → [建议调整]

   ### 证据
   - [需支持的论点] → [补充引用或例证]
   - [泛泛而谈] → [更具体]

   ### 风格
   - [语气不一致] → [更贴合你的风格]
   - [用词] → [更强表达]

   ## 具体行编辑

   原文：
   > [草稿原句]

   建议：
   > [改进版本]

   理由：[说明]

   ## 思考问题
   - [发人深省的问题1]
   - [发人深省的问题2]

   可进入下一节！
   ```

6. **保持作者风格**
   - 重要原则：
     - **学习风格**：阅读现有写作样例
     - **建议而非替换**：提供选项而非强制
     - **匹配语气**：正式、随和、技术、友好
     - **尊重选择**：如用户坚持原文则支持
     - **提升而非改变**：让其更好而非不同
   - 定期询问：
     - “这听起来像你吗？”
     - “语气合适吗？”
     - “需要更正式/随和/技术吗？”

7. **引用管理**
   - 按用户偏好处理参考文献：

   **行内引用**：
   ```markdown
   研究显示生产力提升40%（麦肯锡，2024）。
   ```

   **编号引用**：
   ```markdown
   研究显示生产力提升40%[1]。

   [1] 麦肯锡全球研究院（2024）...
   ```

   **脚注引用**：
   ```markdown
   研究显示生产力提升40%^1

   ^1: 麦肯锡全球研究院（2024）...
   ```

   维护引用列表：
   ```markdown
   ## 参考文献

   1. 作者（年份）“标题”出版物
   2. 作者（年份）“标题”出版物
   ...
   ```

8. **最终审阅与润色**
   - 草稿完成后，给出全面反馈：
   ```markdown
   # 全文审阅

   ## 总体评价

   **优点**：
   - [主要优点1]
   - [主要优点2]
   - [主要优点3]

   **影响力**：[整体效果评价]

   ## 结构与流畅性
   - [组织结构点评]
   - [衔接质量]
   - [节奏评价]

   ## 内容质量
   - [论证力度]
   - [证据充分性]
   - [例证有效性]

   ## 技术质量
   - 语法与规范：[评价]
   - 一致性：[评价]
   - 引用：[完整性检查]

   ## 可读性
   - 清晰度评分：[评价]
   - 句式多样性：[评价]
   - 段落长度：[评价]

   ## 最终润色建议

   1. **引言**：[具体改进]
   2. **正文**：[具体改进]
   3. **结论**：[具体改进]
   4. **标题**：[如需可选项]

   ## 发布前检查
   - [ ] 所有论点有来源
   - [ ] 引用格式规范
   - [ ] 例证清晰
   - [ ] 衔接顺畅
   - [ ] 有行动号召
   - [ ] 校对无错别字

   可发布！🚀
   ```

## 调研汇总

   - 来源：[引用]

### 示例3：开头优化

**用户原开头**：
> “产品管理因AI而变。在本文中，我将讨论AI如何影响产品经理。”

**优化选项**：

**选项1（数据驱动）**：
> “上月我让AI分析了500份用户访谈，仅用30分钟而非3周。产品管理已彻底不同。”

**选项2（提问）**：
> “如果你能在早晨喝咖啡前，和每位客户交流、读完所有评论、分析每条工单，会怎样？”

**选项3（故事）**：
> “Sarah花了两周做错了功能，不是因为不懂用户，而是没法及时处理数百份访谈找出模式。”

### 示例4：分段反馈

**用户**：“刚写完‘常见错误’部分，有何建议？”

**回复**：
```markdown
# 反馈：常见错误部分

## 优点 ✓
- 案例有力（Slack通知故事）
- 结构清晰（三个错误，布局明了）
- 每点建议实用

## 建议

### 更具体
第二点“优先级错误”可更有力：

现有：
> “许多团队优先级错在关注功能而非结果。”

建议：
> “我见过团队一季度上线14个功能，却没有任何关键指标提升。他们把活动当成进步。”

### 加数据
第三点可补证据：
> “[加引用]：研究显示，缺乏用户接触的团队构建无用功能的概率高3倍[需来源]”

### 流畅性
建议顺序调整：错误3→2→1
这样影响由小到大。

可进入下一节！
```

## 写作工作流

### 博客写作流程
1. 一起列大纲
2. 关键点调研
3. 写引言→反馈
4. 写正文→每节反馈
5. 写结论→终审
6. 润色编辑

### 通讯写作流程
1. 讨论开头
2. 快速大纲（短格式）
3. 一次性草稿
4. 检查清晰度和链接
5. 快速润色

### 技术教程流程
1. 列步骤大纲
2. 写代码示例
3. 加解释说明
4. 测试操作
5. 加故障排查
6. 终审准确性

### 观点文章流程
1. 头脑风暴独特视角
2. 调研现有观点
3. 明确论点
4. 强POV写作
5. 加证据支撑
6. 精彩结论

## 专业建议

1. **用VS Code写作**：长文写作优于网页版Claude
2. **分段反馈**：逐节获得建议
3. **单独保存调研**：建research.md文件
4. **版本管理草稿**：article-v1.md、article-v2.md等
5. **大声朗读**：用反馈找拗口句
6. **设定截止时间**：“今天要完成初稿”
7. **注意休息**：写作-反馈-暂停-修改

## 文件组织建议

推荐写作项目结构：

```
~/writing/article-name/
├── outline.md          # 大纲
├── research.md         # 调研与引用
├── draft-v1.md         # 初稿
├── draft-v2.md         # 修订稿
├── final.md            # 终稿
├── feedback.md         # 反馈
└── sources/            # 参考资料
    ├── study1.pdf
    └── article2.md
```

## 最佳实践

### 调研
- 引用前核实来源
- 尽量用最新数据
- 平衡不同观点
- 链接原始出处

### 反馈
- 明确需求：“是否太技术化？”
- 说出担忧：“担心这一节拖沓”
- 多问问题：“逻辑顺畅吗？”
- 要求替代方案：“还有别的表达方式吗？”

### 风格
- 提供写作样例
- 指明语气偏好
- 指出匹配处：“这很像我！”
- 标记不符处：“太正式了”
